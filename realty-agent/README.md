# realty-agent — агент поиска коммерческой недвижимости (Санкт-Петербург)

Инструмент для поиска коммерческой недвижимости в СПб: собирает объявления из
множества источников (классифайды, спец-порталы, базы брокеров и УК, закрытые
каналы), приводит к единой схеме и фильтрует под конкретную цель по параметрам
**гео · площадь/метраж · назначение · этажность · тип операции · бюджет**.

## Статус

- ✅ **Этап 1 (сделано):** каталог источников + спецификация данных и фильтра.
  - [`SOURCES.md`](./SOURCES.md) — исчерпывающий реестр источников с пометкой
    открытый/закрытый и **как считывается** каждый (режимы A/B/C).
  - [`FILTER_SPEC.md`](./FILTER_SPEC.md) — нормализованная схема объекта,
    параметры запроса, справочник назначений и правила matching «объект ↔ цель».
- ⏳ **Этап 2 (спроектировано, код позже):** Cloudflare Worker — сбор по cron,
  хранение в KV, HTTP-эндпоинт `/search` с фильтрацией. См. «Roadmap» ниже.

## Как считываются источники (кратко)

| Режим | Что это | Чем читаем |
|-------|---------|-----------|
| **A** | официальный API / открытый фид | обычный `fetch` (+ ключ API) |
| **B** | открытый сайт с антиботом/JS | **ScrapingBee** (наш «webee»-токен) |
| **C** | закрытый канал/база | Telegram API / ручной экспорт / партнёрский доступ |

Подробности по каждому источнику и его режиму — в [`SOURCES.md`](./SOURCES.md).

> ⚠️ **Про «webee».** В задаче «токены webee для считывания» трактованы как
> **ScrapingBee** (`scrapingbee.com`) — API, который рендерит страницы с антиботом
> и возвращает готовый HTML/JSON по API-ключу. Если имелся в виду другой сервис —
> замену делать в одном месте (адаптер режима B), интерфейс не меняется.

> ⚠️ **Правовое.** Парсинг чужих классифайдов (Авито/ЦИАН/Яндекс/Домклик) нарушает
> их ToS. Приоритет — официальным API и **прямым базам брокеров/УК**, которые
> публикуют свои объекты сами. Классифайды — осознанно и с лимитом частоты.

## Что понадобится для Этапа 2

| Секрет / ресурс | Зачем | Как задать |
|-----------------|-------|-----------|
| `SCRAPINGBEE_KEY` | считывание источников режима B | `wrangler secret put SCRAPINGBEE_KEY` |
| `YANDEX_GEOCODER_KEY` | гео-нормализация (район/метро/радиус) | `wrangler secret put ...` |
| `TELEGRAM_API_*` (опц.) | публичные ТГ-каналы (режим C) | `wrangler secret put ...` |
| KV namespace `REALTY_KV` | хранение собранных объектов | `wrangler kv namespace create REALTY_KV` |

Секреты — только через `wrangler secret put` или дашборд Cloudflare, **не в коде и
не в `wrangler.jsonc`** (конвенция репозитория, см. `brief-bot/README.md`).

## Roadmap Этапа 2 (Cloudflare Worker)

Скелет — по образцу `../lead-worker/worker.js` и `../brief-bot/worker.js`:

```
realty-agent/
├── worker.js         # export default { fetch, scheduled }
├── wrangler.jsonc    # name, main, compatibility_date, KV binding, triggers.crons
├── package.json      # wrangler ^4, type: module, scripts: dev/deploy
├── sources/          # адаптер на источник: { fetch(params)->raw, parse(raw)->Listing[] }
├── SOURCES.md        # (готово)
└── FILTER_SPEC.md    # (готово)
```

- `scheduled` (cron) — обход источников режимов A/B, нормализация в схему
  `Listing` (`FILTER_SPEC.md` §1), запись в `REALTY_KV`.
- `fetch` — `POST /search` с телом `SearchQuery` (`FILTER_SPEC.md` §2) →
  отфильтрованный и ранжированный список (`filterAndRank`, §6).
- Адаптеры источников изолированы: единый интерфейс, режим (A/B/C) выбирается
  внутри адаптера. HTML-парсинг — через `HTMLRewriter` (встроен в Worker) или
  structured-data ответ ScrapingBee, без внешних библиотек.
- Приоритет подключения: брокеры/УК → спец-порталы → классифайды → закрытые каналы.

### Проверка Этапа 2 (когда дойдём)

`wrangler dev`, затем `POST /search` с тестовым запросом, напр.:

```json
{ "operation": "rent", "purpose": "catering",
  "geo": { "districts": ["Центральный"] },
  "areaMin": 80, "areaMax": 130, "floorMin": 1, "floorMax": 1,
  "requireFeatures": ["wetPoints", "ventilation"], "strictness": "balanced" }
```

Ожидаемо: из сохранённых в KV объектов возвращаются только помещения под общепит
80–130 м² на 1 этаже в Центральном районе; объекты без подтверждённых мокрых
точек/вентиляции помечены `needsVerification` и стоят ниже в ранге.
